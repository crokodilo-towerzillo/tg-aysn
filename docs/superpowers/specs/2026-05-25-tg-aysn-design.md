# Дизайн: Telegram-бот расчёта налога АУСН для ВБ-селлеров

**Дата:** 2026-05-25  
**Статус:** в разработке

---

## Контекст

Telegram-бот для ИП/ООО на АУСН «доходы» (8%), торгующих на Wildberries. Бот получает данные из еженедельных финансовых отчётов WB через API ключ и вычисляет налоговую базу за выбранный период.

---

## Бизнес-правила

- Налоговая ставка: **8% от доходов** (АУСН, объект «доходы»)
- База: столбец **продажи** из еженедельных финансовых отчётов WB (возвраты уже учтены в нём)
- Привязка недели к месяцу: неделя входит в месяц, если **дата окончания отчётного периода** попадает в этот месяц
- Результат: только **сумма налога** (база × 8%), округление: копейки вверх до рубля (`math.ceil`)

---

## Функциональность

### Управление API ключами
- Добавить ключ с меткой (названием магазина)
- Просмотреть список своих ключей
- Удалить ключ

У каждого пользователя (Telegram `chat_id`) — свой изолированный список ключей.

### Расчёт налога
1. Выбрать магазин (если ключей несколько — список кнопок)
2. Выбрать период:
   - **Этот месяц** — текущий календарный месяц
   - **Прошлый месяц** — предыдущий календарный месяц
   - **Свой период** — ввод в формате `MM.YY-MM.YY` (например, `01.26-03.26` = январь–март 2026 включительно)
3. Бот запрашивает отчёты у WB API и возвращает: **налог к уплате = Х руб.**

## Архитектура

### Стек
- Python 3.11+
- aiogram 3 (Telegram-бот, FSM)
- SQLite (хранение ключей)
- httpx (async HTTP-запросы к WB API)
- cryptography / Fernet (шифрование ключей)
- python-dotenv (.env конфиг)

### Структура файлов

```
tg-aysn/
├── bot.py            # точка входа, polling, регистрация роутеров
├── config.py         # .env: BOT_TOKEN, FERNET_KEY
├── db.py             # SQLite + Fernet: CRUD с прозрачным шифрованием
├── calculator.py     # WB API + фильтрация недель + расчёт 8%
├── routers/
│   ├── keys.py       # FSM: добавить / удалить / список ключей
│   └── calc.py       # выбор ключа → период → результат
└── keyboards.py      # все inline-клавиатуры
```

### База данных

```sql
CREATE TABLE api_keys (
    id       INTEGER PRIMARY KEY,
    user_id  INTEGER NOT NULL,  -- Telegram chat_id
    label    TEXT    NOT NULL,  -- метка магазина
    key_enc  BLOB    NOT NULL   -- Fernet-зашифрованный WB API ключ
);
```

### Шифрование ключей
- `FERNET_KEY` генерируется один раз (`Fernet.generate_key()`), хранится в `.env`
- Шифрование при записи в БД, расшифровка при чтении — инкапсулировано в `db.py`
- Накладные расходы на шифрование: микросекунды (незаметно на фоне WB API)

### WB API
- Эндпоинт: `GET /api/v1/supplier/reportDetailByPeriod`
- Параметры: `dateFrom`, `dateTo`
- Объём: 1 неделя = 2 строки → за месяц ~8 строк, за 5 лет ~520 — пагинация не нужна
- Фильтр: строки где дата окончания недели (`rr_dt`) попадает в выбранный период
- Суммирование: поле уточняется опытным путём при разработке
- Конкретное поле и оптимальная частота запросов подбираются по Swagger-файлам из предыдущего проекта

## UX-флоу

### Главный экран (/start и /keys)

Экран адаптируется под наличие ключей у пользователя:

**Новый пользователь (нет ключей):**
```
Ваши магазины:

[ Добавить ключ ]
```

**Существующий пользователь (есть ключи):**
```
Ваши магазины:

[ 🟢 Магазин 1 ]
[ 🔴 Магазин 2 ]
[ Добавить ключ ]
[ Удалить ключ  ]
```

Кнопки магазинов — это и есть запуск расчёта для этого ключа.  
«Удалить ключ» появляется только если есть хотя бы один ключ.

**Индикатор статуса ключа** проверяется при каждом `/start`: лёгкий запрос к WB API на каждый ключ.
- 🟢 — ключ рабочий
- 🔴 — ключ недействителен (401); расчёт по нему недоступен, пользователь удаляет и добавляет новый

### FSM: общие правила
- В каждом FSM-диалоге есть кнопка [Отмена] → сброс состояния → главный экран
- `/start` в любом состоянии FSM → сброс состояния → главный экран
- Если пользователь закрыл Telegram и открыл снова (бот работает) — FSM-состояние сохранено в памяти, бот всё ещё ждёт ввода; `/start` выводит на главный экран

### Добавление ключа
```
[Добавить ключ]
→ «Введите метку магазина» + [Отмена] (FSM: ожидание текста)
→ «Введите API ключ WB» + [Отмена] (FSM: ожидание текста)
→ Проверка ключа запросом к WB API
  → 401: «Ключ недействителен, попробуйте другой» → остаёмся в FSM
  → OK: «Магазин "Название" добавлен 🟢» → главный экран
```

### Удаление ключа
```
[Удалить ключ]
→ Список кнопок с именами магазинов + [Отмена]
→ Выбор магазина → «Магазин "Название" удалён.» → главный экран
```

### Расчёт налога
```
[Магазин N] (с главного экрана)
→ Выбор периода: [Этот месяц] [Прошлый месяц] [Свой период]
  → Свой период: «Введите период в формате MM.YY-MM.YY» + [Отмена] (FSM)
→ Запрос к WB API...
→ «Налог АУСН 8%: 12 345 руб.»
  [ Главная ]
```

Кнопка [Главная] также доступна на экране выбора периода.

## Обработка ошибок

- Неверный WB API ключ (401) при расчёте → «Ключ недействителен, удалите и добавьте новый»
- Нет данных за период → «За выбранный период отчётов не найдено»
- Неверный формат периода → «Формат: MM.YY-MM.YY, например 01.26-03.26»
- Таймаут / ошибка WB API → «WB не ответил, попробуйте позже»
- Попытка расчёта по красному ключу 🔴 → «Ключ недействителен, обновите его»
