# Дизайн: Telegram-бот расчёта налога АУСН для ВБ-селлеров

**Дата:** 2026-05-25  
**Статус:** в разработке

---

## Контекст

Telegram-бот для ИП/ООО на АУСН «доходы» (8%), торгующих на Wildberries. Бот получает данные из еженедельных финансовых отчётов WB через API ключ и вычисляет налоговую базу за выбранный период.

---

## Бизнес-правила

- Налоговая ставка: **8% от доходов** (АУСН, объект «доходы»)
- База: поле `retailAmountSum` из отчётов реализации WB (`/sales-reports/list`), суммируется по всем типам отчётов за период
- Привязка недели к месяцу: неделя входит в месяц, если **дата окончания отчётного периода** попадает в этот месяц
- Результат: только **сумма налога** (база × 8%), округление: копейки вверх до рубля (`math.ceil`)

---

## Функциональность

### Управление API ключами
- Добавить ключ с меткой (названием магазина)
- Просмотреть список своих ключей
- Удалить ключ

У каждого пользователя (Telegram `chat_id`) — свой изолированный список ключей.

### Расчёт налога
1. Выбрать магазин (если ключей несколько — список кнопок)
2. Выбрать период:
   - **Этот месяц** — текущий календарный месяц
   - **Прошлый месяц** — предыдущий календарный месяц
   - **Свой период** — ввод в формате `MM.YY-MM.YY` (например, `01.26-03.26` = январь–март 2026 включительно)
3. Бот считает налог из локального кеша и возвращает: **налог к уплате = Х руб.**

### База данных

```sql
CREATE TABLE api_keys (
    id             INTEGER PRIMARY KEY,
    user_id        INTEGER NOT NULL,   -- Telegram chat_id
    label          TEXT    NOT NULL,   -- метка магазина
    key_enc        BLOB    NOT NULL,   -- Fernet-зашифрованный WB API ключ
    last_synced_at TEXT                -- ISO datetime последней синхронизации отчётов
);

CREATE TABLE reports (
    id                INTEGER PRIMARY KEY,
    key_id            INTEGER NOT NULL REFERENCES api_keys(id) ON DELETE CASCADE,
    report_id         INTEGER NOT NULL,  -- ID отчёта от WB
    date_from         TEXT    NOT NULL,  -- начало периода отчёта
    date_to           TEXT    NOT NULL,  -- конец периода отчёта (по нему фильтруем месяц)
    report_type       INTEGER NOT NULL,  -- 1 основной, 2 по выкупам, 3 по выкупам (Грузия)
    retail_amount_sum REAL    NOT NULL,  -- retailAmountSum — налоговая база
    UNIQUE(key_id, report_id)
);
```

### Шифрование ключей
- `FERNET_KEY` генерируется один раз (`Fernet.generate_key()`), хранится в `.env`
- Шифрование при записи в БД, расшифровка при чтении — инкапсулировано в `db.py`
- Накладные расходы на шифрование: микросекунды (незаметно на фоне WB API)

### WB API
- Хост: `https://finance-api.wildberries.ru`
- Эндпоинт: `POST /api/finance/v1/sales-reports/list`
- Токен: Персональный или Сервисный, категория **Финансы**
- Rate limit: 1 запрос / 1 мин
- Данные доступны: с **1 января 2025** — нижняя граница периода не раньше этой даты
- Тело запроса: `{"dateFrom": "YYYY-MM-DD", "dateTo": "YYYY-MM-DD"}`
- Поле налоговой базы: `retailAmountSum` («Продажа») — в ответе тип `string`, при сохранении явное преобразование `float()`
- Суммировать по всем `reportType` (1 — основной, 2 — по выкупам, 3 — по выкупам для Грузии)
- Фильтр: отчёты где `dateTo` попадает в выбранный период
- Неделя на границе месяцев принадлежит тому месяцу, в который попадает `dateTo` — без исключений
- Пагинация не нужна (за месяц ~4-5 отчётов)

**Валидация ключей** (при добавлении и при `/start`):
- Эндпоинт: `GET /api/v1/account/balance` (отдельный rate-limit бюджет, не сжигает слот финансовых отчётов)
- 200 → ключ рабочий 🟢, 401 → недействителен 🔴

## UX-флоу

### Главный экран (/start и /keys)

Экран адаптируется под наличие ключей у пользователя:

**Новый пользователь (нет ключей):**
```
Ваши магазины:

[ Добавить ключ ]
```

**Существующий пользователь (есть ключи):**
```
Ваши магазины:

[ 🟢 Магазин 1 ]
[ 🔴 Магазин 2 ]
[ Добавить ключ ]
[ Удалить ключ  ]
```

Кнопки магазинов — это и есть запуск расчёта для этого ключа.  
«Удалить ключ» появляется только если есть хотя бы один ключ.

**При каждом `/start`** выполняется синхронизация с cooldown:
- Если `last_synced_at` < 10 минут назад — синхронизация пропускается, сразу главный экран
- Иначе:
  1. Бот отправляет «Обновляю данные...»
  2. Для каждого ключа параллельно (`asyncio.gather`): запрос `/sales-reports/list` с `dateFrom = last_synced_at` (или 2025-01-01 при первой загрузке), `dateTo = сегодня`
  3. Новые отчёты добавляются в таблицу `reports` (UPSERT по `key_id + report_id`)
  4. `last_synced_at` обновляется
  5. Сообщение обновляется на главный экран

**Защита от гонки при удалении:** перед записью в `reports` проверяем что `key_id` ещё существует в `api_keys`; если нет — прерываем корутину без ошибки.

- 🟢 — синхронизация прошла успешно
- 🔴 — запрос вернул 401; расчёт по ключу недоступен, пользователь удаляет и добавляет новый

Все расчёты выполняются **из локальной БД** — без дополнительных запросов к WB API.

### FSM: общие правила
- В каждом FSM-диалоге есть кнопка [Отмена] → сброс состояния → главный экран
- `/start` в любом состоянии FSM → сброс состояния → главный экран
- Если пользователь закрыл Telegram и открыл снова (бот работает) — FSM-состояние сохранено в памяти, бот всё ещё ждёт ввода; `/start` выводит на главный экран
- Если бот перезапустился (MemoryStorage сброшен) — любой входящий callback без FSM-состояния обрабатывается как `/start`

### Добавление ключа
```
[Добавить ключ]
→ «Введите метку магазина» + [Отмена] (FSM: ожидание текста)
→ «Введите API ключ WB» + [Отмена] (FSM: ожидание текста)
   Подсказка: «Создайте токен в ЛК WB → Профиль → Настройки →
   Доступ к API → категория Финансы (тип: Персональный или Сервисный)»
→ Валидация через GET /api/v1/account/balance
  → 401: «Ключ недействителен, проверьте тип токена и попробуйте снова» → остаёмся в FSM
  → OK: ключ сохраняется, «Загружаю историю отчётов...»
      → POST /sales-reports/list (dateFrom=2025-01-01, dateTo=сегодня)
      → отчёты сохраняются в таблицу reports
      → «Магазин "Название" добавлен 🟢» → главный экран
```

### Удаление ключа
```
[Удалить ключ]
→ Список кнопок с именами магазинов + [Отмена]
→ Выбор магазина → «Удалить "Название"? Данные отчётов будут удалены.»
  [ Да, удалить ]  [ Отмена ]
→ [Да, удалить] → «Магазин "Название" удалён.» → главный экран
→ [Отмена] → главный экран
```

### Расчёт налога
```
[Магазин N] (с главного экрана)
→ Выбор периода: [Этот месяц] [Прошлый месяц] [Свой период] [Главная]
  → Свой период: «Введите период в формате MM.YY-MM.YY
                  Например: 01.26-03.26 (январь–март 2026)» + [Отмена] (FSM)
    → Валидация: месяц 01–12, конец не раньше начала
    → Ошибка: «Неверный формат. Введите MM.YY-MM.YY, например 01.26-03.26» → повтор ввода
→ Расчёт из локальной БД (мгновенно)
→ «Налог АУСН 8% за январь–март 2026: 12 345 руб.»
  [ Другой период ]  [ Главная ]
```

В результате всегда показывается итоговый период словами (январь 2026 / январь–март 2026).
[Другой период] возвращает к выбору периода для того же магазина.

## Обработка ошибок

- Неверный WB API ключ (401) при расчёте → «Ключ недействителен, удалите и добавьте новый»
- Нет данных за период → «За выбранный период отчётов не найдено» + [Другой период] [Главная]
- Неверный формат периода → «Неверный формат. Введите MM.YY-MM.YY, например 01.26-03.26» → повтор ввода
- Конец периода раньше начала → «Конец периода не может быть раньше начала» → повтор ввода
- Период раньше января 2025 → «Данные доступны с января 2025 года» → повтор ввода
- Превышен rate limit WB API (429) при синхронизации → ключ помечается 🔴 временно, сообщение «WB ограничил запросы, попробуйте позже» (не удалять ключ)
- Таймаут WB API (>5 сек) → «WB не ответил, попробуйте позже»
- WB вернул пустой ответ или невалидный JSON → «Не удалось получить данные, попробуйте позже»
- Сетевая ошибка (ConnectionError) → «Не удалось получить данные, попробуйте позже»
- Попытка расчёта по красному ключу 🔴 → «Ключ недействителен, обновите его»
